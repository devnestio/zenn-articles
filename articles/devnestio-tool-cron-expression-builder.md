## 作ったもの

開発者向けの無料ツール群「devnestio」に、**Cron Expression Builder** を追加しました。`*/15 * * * *` のようなcron式を入力すると、その意味を人間が読める形（「15分ごと」など）で説明し、さらに**次回の実行時刻を何件か先まで計算して並べて表示する**Webツールです。逆に、フィールドをGUIで組み立ててcron式を生成することもできます。

👉 https://cron-expression-builder.pages.dev

外部ライブラリゼロ・単一HTMLファイル・サーバー送信なしという、devnestioの共通方針に沿って作っています。

## 作った動機

cron式は、書く頻度が低いわりに毎回うっすら不安になる記法の代表格だと思っています。`30 2 * * 1` が「毎週月曜の2:30」なのか「毎月1日の…」なのか、フィールドの順番（分・時・日・月・曜日）をその場で思い出せない。`*/5` と `0/5` の違い、曜日の `0` が日曜なのか、といった細かい仕様も毎回ググることになります。

そして一番こわいのは、**式が間違っていても、実際にその時刻が来るまで気づけない**ことです。デプロイしてから「あれ、動いてない」となるのが一番コストが高い。なので「この式は次にいつ動くのか」を**書いた瞬間に確認できる**ツールがほしかった、というのが動機です。

## 主な機能

- **双方向変換** — cron式 → 説明文、フィールドGUI → cron式 の両方向
- **次回実行時刻のプレビュー** — 直近の実行予定を複数件、日時で一覧表示
- **人間語での説明** — 「毎時0分」「平日の9時」のように自然文へ展開
- **よく使うプリセット** — 毎分・毎時・毎日・毎週・毎月などをワンクリック挿入
- **完全オフライン動作** — 計算はすべてブラウザ内、ネットワーク送信なし

## 技術的なポイント：パースより「次回時刻の計算」が本体

cron式そのもののパースは、実はそれほど難しくありません。5つのフィールドを空白で分割し、各フィールドを `*` / 数値 / `,`（リスト）/ `-`（範囲）/ `/`（ステップ）に応じて「許可される値の集合」へ展開するだけです。たとえば分フィールドを 0〜59 の配列に対して評価していくイメージです。

```js
// 1フィールドを許可値の集合に展開する
function expandField(expr, min, max) {
  const allowed = new Set();
  for (const part of expr.split(",")) {
    const [range, step] = part.split("/");
    const s = step ? +step : 1;
    let lo = min, hi = max;
    if (range !== "*") {
      const [a, b] = range.split("-");
      lo = +a; hi = b !== undefined ? +b : +a;
    }
    for (let v = lo; v <= hi; v += s) allowed.add(v);
  }
  return allowed;
}
```

本体はむしろ**「次回実行時刻の計算」**のほうです。素朴にやるなら、現在時刻から1分ずつ未来に進め、各フィールドの許可集合にすべてマッチするかを判定していく、という総当たりになります。これは実装が単純で間違いにくい代わりに、`0 0 29 2 *`（2月29日）のような滅多に来ない条件だと、何年も先までループが回ってしまいます。

なので、**分→時→日→月の順に、合わない桁を一気に繰り上げる**やり方にしました。たとえば「時」が合わなければ分を0に戻して時を1進める、というふうに桁ごとにスキップすると、無駄な反復を大幅に減らせます。あわせて、探索が一定回数（=数年分）を超えたら「該当なし」と打ち切るガードも入れています。`30 2 31 2 *`（2月31日＝永遠に来ない）のような不正条件で無限ループに陥らないためです。

cronの厄介な仕様として、**「日」フィールドと「曜日」フィールドはOR条件**（どちらかが合えば実行）という伝統的な挙動があります。ここをAND条件と勘違いすると結果がズレるので、判定ロジックで明示的に分岐させています。

## フレームワークは使わない

状態としては「現在の式」と「計算した次回時刻リスト」くらいしかなく、Reactを持ち込むほどの複雑さはありません。入力に対して関数を一本通して結果を描き直すだけなので、単一HTMLに素のJSを書くほうが読みやすく落ちにくい、というのがdevnestio全体での結論です。Cloudflare Pagesに静的ファイルとして置くだけでデプロイが終わるのも、この構成の利点です。

## 実装はAIエージェントに任せた

devnestioのツールは、Claude Code sessions に発注書（PRD）を渡して自律的に実装してもらう形で作っています。今回も「5フィールドのパース、次回実行時刻の計算、日と曜日はOR条件、不正式での打ち切りガード」という要件を渡し、実装からテストまでをエージェント側で完結させました。

効いたのは、発注書に**ハマりどころと正解を先回りで書いておいた**ことです。具体的には「日と曜日はAND ではなく OR」「`0` は日曜」「来ない条件（2月31日など）は無限ループせず打ち切る」「`*/15` のステップ起点は0」を明記しました。cronは直感に反する仕様が多いので、こうした仕様を曖昧に渡すと「それっぽいけど微妙に間違ったツール」が出てきがちです。判定基準を具体的に固定すると品質が安定します。

## テスト結果

ブラウザ実行型のテストハーネス（test.html）で **38/38 PASS**。パース結果（各フィールドの許可集合）と、代表的な式の次回実行時刻を期待値と照合しています。`*/15 * * * *`（15分ごと）、`0 9 * * 1-5`（平日9時）、`30 2 * * 0`（日曜2:30）に加え、`0 0 29 2 *`（うるう年の2月29日）が正しく数年先まで飛ぶこと、`30 2 31 2 *`（来ない条件）が打ち切られて該当なしになることまで確認しました。境界条件（月末、年またぎ、曜日のOR判定）を重点的に見ています。
