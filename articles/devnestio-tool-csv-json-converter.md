CSV と JSON を**双方向**で変換できるツールです。スプレッドシートからコピペした CSV を JSON 配列に、API レスポンスの JSON を Excel で開ける CSV に、どちらの向きにもワンクリックで変換できます。外部ライブラリゼロ・単一 HTML ファイル・サーバー送信なし（すべてブラウザ内で完結）という devnestio の共通方針に沿って実装しています。

## 作った動機

「CSV と JSON を行き来したい」という場面は地味に多いです。

- DB からエクスポートした CSV を、フロントのモックデータ（JSON）に変換したい
- API から返ってきた JSON 配列を、Excel や Google Sheets で開いて非エンジニアに共有したい
- スプレッドシートで管理しているマスタデータを、そのまま設定ファイル（JSON）に流し込みたい

既存のオンライン変換ツールは便利なのですが、**データをサーバーに送信するもの**が多く、顧客データや社内マスタを貼り付けるのはためらいます。かといって Python で `csv` と `json` を書くのは、毎回やるには面倒。「ブラウザのタブ1枚で、貼って→押すだけ」で完結するものが欲しい、というのが動機でした。

## 主な機能

### 1. CSV → JSON（オブジェクト配列）

1行目をヘッダーとして扱い、各行をオブジェクトに変換します。

```
# 入力（CSV）
id,name,active
1,Alice,true
2,Bob,false
```

```json
// 出力（JSON）
[
  { "id": 1, "name": "Alice", "active": true },
  { "id": 2, "name": "Bob", "active": false }
]
```

`1` は数値、`true`/`false` は真偽値として自動で型推論します（後述）。

### 2. JSON → CSV

オブジェクトの配列を CSV に戻します。キーの和集合をヘッダーに展開するので、行ごとにキーが欠けていても列がズレません。

```json
[
  { "id": 1, "name": "Alice" },
  { "id": 2, "name": "Bob", "city": "Tokyo" }
]
```

```
id,name,city
1,Alice,
2,Bob,Tokyo
```

### 3. 区切り文字の切り替え

カンマだけでなく、**タブ区切り（TSV）・セミコロン区切り**にも対応。ヨーロッパ圏の CSV（小数点がカンマなのでセミコロン区切りが主流）や、スプレッドシートからのコピペ（タブ区切りになる）をそのまま扱えます。

### 4. クォート・エスケープの正しい処理

CSV の鬼門である「値の中のカンマ」「値の中の改行」「ダブルクォートのエスケープ（`""`）」を RFC 4180 準拠で処理します。

```
name,note
"Smith, John","He said ""hi"""
```

これを素朴に `split(',')` すると壊れますが、ステートマシンでパースしているので正しく1行2フィールドとして解釈されます。

## 技術的なポイント：型推論とクォート処理

### 型推論をどこまでやるか

CSV はすべて文字列です。これを JSON にするとき、`"123"` を数値 `123` にするか文字列のままにするかは悩みどころです。devnestio 版では次のルールにしました。

- 数値として解釈できる（`^-?\d+(\.\d+)?$`）→ number
- `true` / `false`（大文字小文字無視）→ boolean
- 空文字 → `null` ではなく空文字列 `""`（情報を失わないため）
- それ以外 → string

「`007` のような先頭ゼロは数値化しない」など、データを壊さない方向に倒しています。郵便番号や ID が勝手に数値化されて先頭ゼロが消える、という事故を避けるためです。

### CSV パーサーはステートマシンで実装

`split(',')` ではクォート内のカンマに対応できないため、1文字ずつ走査するステートマシンで実装しています。

```javascript
// 概念コード（実際の実装を簡略化）
function parseCSV(text, delimiter) {
  const rows = [];
  let field = '', row = [], inQuotes = false;
  for (let i = 0; i < text.length; i++) {
    const c = text[i];
    if (inQuotes) {
      if (c === '"' && text[i + 1] === '"') { field += '"'; i++; }
      else if (c === '"') { inQuotes = false; }
      else { field += c; }
    } else {
      if (c === '"') inQuotes = true;
      else if (c === delimiter) { row.push(field); field = ''; }
      else if (c === '\n') { row.push(field); rows.push(row); field = ''; row = []; }
      else if (c !== '\r') field += c;
    }
  }
  if (field !== '' || row.length) { row.push(field); rows.push(row); }
  return rows;
}
```

「クォート内かどうか」のフラグ1つで、カンマ・改行・エスケープを正しく捌けます。地味ですが、ここを手を抜くと実データで必ず壊れます。

### なぜ外部ライブラリを使わないか

`papaparse` を使えば一瞬ですが、devnestio では「単一 HTML ファイル・外部依存ゼロ」をポリシーにしています。理由は、ユーザーが貼り付けたデータが第三者スクリプトに渡る経路を完全に断つこと、そして初回アクセス以降はブラウザキャッシュで**完全オフライン動作**させることの2点です。社内データを扱うときに「このツールは外に何も送らない」と断言できるのは強いです。

## 実装はAIエージェントに任せた

devnestio のツールは、Claude Code sessions に発注書（PRD）を渡して自律的に実装してもらう形で作っています。

今回特に効いたのは、発注書に「RFC 4180 準拠でクォート・エスケープ・埋め込み改行を処理すること」「型推論のルールを明示（先頭ゼロは数値化しない等）」「papaparse 等の外部ライブラリは使用禁止」と**仕様の境界条件を具体的に書いた**ことです。CSV は「だいたい動く」実装が一番危険な領域なので、エッジケースを発注書側で潰しておくと、出来上がりの品質が大きく変わります。

**テスト結果:** ブラウザ実行型テスト（test.html）で **120/120 PASS**

埋め込みカンマ・埋め込み改行・エスケープされたクォート・空フィールド・不揃いなキー・TSV/セミコロン区切りなど、壊れやすいパターンを重点的にテストケース化しました。
