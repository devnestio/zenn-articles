## 作ったもの

開発者向けの無料ツール群「devnestio」に、**Hash Generator** を追加しました。テキストを貼り付けると、MD5 / SHA-1 / SHA-256 / SHA-384 / SHA-512 のハッシュ値を同時に計算して、16進数で並べて表示するWebツールです。チェックサムの確認や、APIの署名生成の下調べみたいな場面でサッと使えます。

👉 https://hash-generator-745.pages.dev

外部ライブラリゼロ・単一HTMLファイル・サーバー送信なし（すべてブラウザ内で完結）という、devnestioの共通方針に沿って作っています。

## 作った動機

「このファイルのSHA-256ちょっと知りたい」「ドキュメントに載っているチェックサムと一致するか確認したい」みたいな用事は、開発をしているとそこそこの頻度で出てきます。ターミナルで `shasum -a 256` を叩けば済むのですが、Windowsだったり、ちょっとした文字列のハッシュを取りたいだけだったりすると、コマンドを思い出すより貼り付けたほうが早い。

ここでも気になるのは**入力をどこに送るか**です。ハッシュ化したい元データ（パスワードのテスト値や内部の識別子など）を他人のサーバーに送るのは避けたい。であれば、ブラウザ内で完結する計算機を持っておけばいい、というのが動機です。ハッシュ計算は本来クライアント側で完結できる処理なので、サーバーに送る理由がそもそもありません。

## 主な機能

- **5アルゴリズム同時計算** — MD5 / SHA-1 / SHA-256 / SHA-384 / SHA-512 を一度に表示
- **リアルタイム更新** — 入力するたびに即座に再計算
- **ワンクリックコピー** — 各ハッシュ値を個別にコピー
- **完全オフライン動作** — 計算はすべてブラウザ内、ネットワーク送信なし

## 技術的なポイント：Web Crypto APIとMD5問題

SHA系のハッシュは、ブラウザ標準の **Web Crypto API**（`crypto.subtle.digest`）で計算できます。外部ライブラリを一切入れずに済むのがありがたいところです。ただし、このAPIは**非同期（Promise）**で、出力は `ArrayBuffer` なので、16進文字列に変換する処理が必要です。

```js
async function sha(algo, text) {
  const data = new TextEncoder().encode(text);
  const buf = await crypto.subtle.digest(algo, data); // "SHA-256" など
  return [...new Uint8Array(buf)]
    .map(b => b.toString(16).padStart(2, "0"))
    .join("");
}
```

ここで `padStart(2, "0")` を忘れると、`0x0a` が `"a"` になってハッシュ全体が壊れます。地味ですが必ず2桁ゼロ埋めが必要です。

そして最大の落とし穴が **MD5** です。Web Crypto API は**MD5をサポートしていません**。セキュリティ的に非推奨だからです。とはいえ、レガシーなチェックサム照合などでMD5が必要な場面はまだ残っています。なので、MD5だけは**自前のJS実装**を用意しました。MD5のアルゴリズム（512ビットブロック分割、4ラウンドのビット演算、リトルエンディアン処理）を素のJSで書く形です。ビットシフトと32ビット演算の符号の扱いでハマりやすく、ここは入念にテストしました。

「SHAは標準APIで楽、MD5だけ手実装」という非対称な構成になったのが、このツールの一番の技術的なポイントです。

## フレームワークは使わない

状態の少ないツールにReactを持ち込むと、ビルド・バンドルサイズ・依存更新のコストのほうが大きくなります。入力欄とハッシュ表示があるだけのツールなら、単一HTMLに素のJSを書くほうが読みやすく落ちにくい、というのがdevnestio全体での結論です。Cloudflare Pagesに静的ファイルとして置くだけでデプロイが終わるのも、この構成の利点です。

## 実装はAIエージェントに任せた

devnestioのツールは、Claude Code sessions に発注書（PRD）を渡して自律的に実装してもらう形で作っています。今回も「SHA系はWeb Crypto API、MD5は非対応なので自前実装、出力は16進ゼロ埋め」という要件を渡し、実装からテストまでをエージェント側で完結させました。

効いたのは、発注書に「MD5はWeb Crypto非対応。自前実装し、既知のテストベクタ（空文字列 → `d41d8cd98f00b204e9800998ecf8427e` など）で必ず検証すること」と**先回りでハマりどころを書いておいた**ことです。MD5の自前実装は微妙な符号バグを埋め込みやすいので、正解値を渡して照合させると品質が安定しました。

## テスト結果

ブラウザ実行型のテストハーネス（test.html）で **42/42 PASS**。各アルゴリズムについて既知のテストベクタ（空文字列、`"abc"`、長い文字列、マルチバイト文字）と照合し、16進ゼロ埋めの桁数、コピー機能、空入力時の挙動まで確認しています。特にMD5は自前実装なので、RFC 1321 のテストベクタとの一致を重点的に検証しました。
