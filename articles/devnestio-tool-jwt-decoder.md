## 作ったもの

開発者向けの無料ツール群「devnestio」に、**JWT Decoder & Inspector** を追加しました。JWT（JSON Web Token）を貼り付けると、ヘッダー・ペイロード・署名の3パートに分解して、中身を整形表示するWebツールです。`exp` や `iat` といったタイムスタンプは人間が読める日時に変換し、有効期限切れかどうかも一目で分かるようにしています。

👉 https://jwt-decoder.pages.dev

外部ライブラリゼロ・単一HTMLファイル・サーバー送信なし（すべてブラウザ内で完結）という、devnestioの共通方針に沿って作っています。

## 作った動機

JWTの中身をちょっと確認したい場面って、認証まわりを触っていると毎日のように出てきます。「このトークン、どのユーザーで発行された？」「`exp` 切れてない？」「`scope` に必要な権限入ってる？」みたいな確認です。

定番のオンラインデコーダーもあるのですが、**認証トークンを他人のサーバーに貼り付けるのは本来かなり気持ちが悪い**行為です。JWTのペイロードはbase64urlエンコードされているだけで暗号化はされていないので、貼り付けた時点で中身（ユーザーID、メール、権限など）は相手に丸見えになり得ます。開発用のダミートークンならともかく、ステージングや本番に近いトークンを外部サイトに送るのは避けたい。

であれば、**完全クライアントサイドで、ネットワークに何も送らないデコーダー**を自分で持っておけばいい、というのが動機です。ページを開いてオフラインにしても動きます。

## 主な機能

- **3パート分解表示** — ヘッダー / ペイロード / 署名をそれぞれ整形して表示
- **タイムスタンプの自動変換** — `exp` `iat` `nbf` をローカル日時に変換
- **有効期限チェック** — `exp` を現在時刻と比較し、期限切れを警告表示
- **アルゴリズム表示** — ヘッダーの `alg`（HS256 / RS256 など）を明示
- **完全オフライン動作** — デコードはすべてブラウザ内、ネットワーク送信なし

## 技術的なポイント：base64urlの罠

JWTの3パートは `.` 区切りで、各パートは **base64url** エンコードです。ここで素直に `atob()` を使うとハマります。base64urlは通常のbase64と違い、`+` → `-`、`/` → `_` に置き換えられ、末尾の `=` パディングも省略されているからです。

そのまま `atob()` に渡すと例外を吐くので、いったん通常のbase64に戻してからデコードします。

```js
function decodeBase64Url(str) {
  // base64url → base64
  let b64 = str.replace(/-/g, "+").replace(/_/g, "/");
  // パディング復元
  while (b64.length % 4) b64 += "=";
  // atob はバイナリ文字列を返すので UTF-8 として復元
  const bin = atob(b64);
  const bytes = Uint8Array.from(bin, c => c.charCodeAt(0));
  return new TextDecoder("utf-8").decode(bytes);
}
```

もうひとつ地味な落とし穴が**マルチバイト文字**です。JWTのペイロードに日本語の名前などが入っていると、`atob()` の結果をそのまま使うと文字化けします。`atob()` はあくまでバイナリ文字列を返すだけなので、上のように `TextDecoder` でUTF-8として明示的にデコードする必要があります。ここを手抜きすると「英語のトークンは通るのに特定のトークンだけ壊れる」という再現性の低いバグになります。

`exp` の扱いも一点注意があります。JWTのタイムスタンプは **Unix秒**（ミリ秒ではない）なので、`new Date(payload.exp * 1000)` と1000倍してからDateに渡します。ここを間違えると1970年が表示されて軽く混乱します。

```js
const exp = payload.exp ? new Date(payload.exp * 1000) : null;
const isExpired = exp ? exp.getTime() < Date.now() : false;
```

## 署名は「検証しない」と決めた

最初に方針として決めたのは、**このツールでは署名検証（verify）をやらない**ことです。理由は単純で、署名を検証するには秘密鍵（HS256）か公開鍵（RS256）が必要で、それをブラウザのフォームに入力させるのは設計として筋が悪いからです。秘密鍵を扱う時点で「軽量で安全なデコーダー」というコンセプトから外れます。

なので役割を**デコード（中身を読む）に振り切り**、「このツールは署名を検証していません」という注意書きをUIに明示しました。verifyが必要なケースはバックエンドのライブラリでやるべきで、ブラウザのデコーダーが担う領域ではない、という線引きです。スコープを欲張らないほうがツールとしては信頼できます。

フレームワークは使っていません。状態の少ないツールにReactを持ち込むと、ビルド・バンドルサイズ・依存更新のコストのほうが大きくなります。単一HTMLに素のJSを書くほうが、読みやすく落ちにくいというのが今のところの結論です。

## 実装はAIエージェントに任せた

devnestioのツールは、Claude Code sessions に発注書（PRD）を渡して自律的に実装してもらう形で作っています。今回も「3パート分解・base64urlデコード・タイムスタンプ変換・期限チェック・署名検証はしない」という要件を渡し、実装からテストまでをエージェント側で完結させました。

効いたのは、最初の発注書に「base64urlの `+/-` `/_` 変換とパディング復元を必ず行うこと」「マルチバイトは `TextDecoder` でUTF-8デコードすること」と**ハマりポイントを先回りして書いておいた**ことです。これを書かずに丸投げすると、英語トークンだけ通って日本語トークンで化ける、という典型的なバグが残りがちでした。「何を作るか」と「どこでハマるか」を人間が先に潰しておくと、エージェントの出力品質が安定します。

## テスト結果

ブラウザ実行型のテストハーネス（test.html）で **31/31 PASS**。base64urlの各種パターン（パディングあり/なし、`-_` を含むトークン）、マルチバイトペイロードの文字化け検証、`exp` 期限切れ/有効の判定、不正なJWT（パートが足りない・base64が壊れている）を貼ったときに例外で落ちずエラー表示できるか、あたりを網羅しています。

## 今後の展望

- 公開鍵（JWKS）を貼り付けての RS256 署名検証オプション（明示的にオンにしたときだけ）
- よく使うクレーム（`iss` `aud` `sub`）のラベル付き表示
- JWTの生成（デバッグ用にペイロードからトークンを作る）

verifyは「やらない」と決めたものの、JWKSを手元で貼って検証したい需要は分かるので、デフォルトオフのオプションとしてなら検討の余地があると思っています。
